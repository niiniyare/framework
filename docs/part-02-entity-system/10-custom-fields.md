# Chapter 10: Custom Fields — Runtime Schema Extension

No ERP framework can anticipate every business requirement at compile time. A Kenyan logistics company needs to track their KRA-required customs classification code. A Nairobi hospital needs to capture insurance provider information on patient invoices. A cooperative bank needs additional KYC fields on the Customer entity.

Custom fields solve this without requiring source code changes, redeployment, or database migrations. Tenants define fields through the admin UI, and those fields are immediately available in forms, API responses, and filter queries.

---

## 10.1. The Custom Field Model

### 10.1.1. How Custom Fields Extend Both System and Custom Entities

Custom fields can be attached to any entity — both system entities (Go struct-backed, SQL columns) and custom entities (JSONB-backed, tenant-defined). The storage mechanism differs:

- **On system entities**: custom fields are stored in a reserved `custom_fields jsonb` column that every system entity carries. The field values are merged into the entity's JSON response alongside the structured SQL columns.

- **On custom entities**: custom fields are additional keys in the same JSONB document that stores all custom entity data.

The API, SDUI layer, and filter DSL treat custom fields identically to system fields from the caller's perspective.

### 10.1.2. Storage — The `custom_fields JSONB` Column Pattern on System Entities

Every system entity generated by Awo includes:

```sql
ALTER TABLE invoices ADD COLUMN custom_fields jsonb NOT NULL DEFAULT '{}';
CREATE INDEX ON invoices USING gin(custom_fields);
```

The GIN index is added automatically at entity generation time. Individual key paths are additionally indexed on demand when a tenant marks a custom field as "indexed" in the admin UI (see §10.4.2).

Custom field values are stored as:
```json
{
  "kra_customs_code": "HS-8471",
  "insurer_name": "Britam",
  "insurer_policy_number": "BTM-2025-00441",
  "requires_pre_auth": true
}
```

The top-level keys are the stable field keys defined in `CustomFieldDef` (not the display labels).

### 10.1.3. The `CustomFieldDef` System Entity — Metadata Table

`CustomFieldDef` is itself a system entity that describes custom field definitions per tenant:

```go
type CustomFieldDef struct {
    ent.Schema
}

func (CustomFieldDef) Fields() []ent.Field {
    return []ent.Field{
        field.String("entity_name").MaxLen(100).NotEmpty().Immutable(),
        field.String("field_key").MaxLen(100).NotEmpty().Immutable(),
        field.String("label").MaxLen(255).NotEmpty(),
        field.String("description").MaxLen(1024).Optional(),
        field.Enum("field_type").
            Values("data", "small_text", "long_text", "int", "float",
                   "currency", "bool", "date", "datetime", "select",
                   "multiselect", "link").
            Immutable(),
        field.JSON("options", []string{}).Optional(),        // for select/multiselect
        field.JSON("link_target", &LinkTarget{}).Optional(), // for link type
        field.Bool("required").Default(false),
        field.Bool("unique").Default(false),
        field.Bool("indexed").Default(false),
        field.Int("sort_order").Default(0),
        field.String("section_label").MaxLen(100).Optional(),
        field.JSON("validation_rules", &ValidationRules{}).Optional(),
        field.JSON("ui_meta", &UIMetadata{}).Optional(),
        field.Time("deprecated_at").Optional().Nillable(),
    }
}

func (CustomFieldDef) Indexes() []ent.Index {
    return []ent.Index{
        index.Fields("entity_name", "field_key").Unique(),
    }
}
```

`CustomFieldDef` records are per-tenant — each tenant defines their own fields for each entity type. A field defined by tenant A is invisible to tenant B.

### 10.1.4. Scope — Custom Fields Are Per-Tenant, Per-Entity

```
Tenant A
  └── Invoice
        ├── kra_customs_code (Data)
        └── insurer_name (Data)

Tenant B
  └── Invoice
        ├── delivery_vehicle_id (Link → Vehicle)
        └── co2_offset_required (Bool)
```

The same `Invoice` entity renders different custom fields for different tenants. The `custom_fields` JSON value in the database also differs per-tenant — there is no cross-contamination because each tenant has their own PostgreSQL schema.

## 10.2. Defining Custom Fields

### 10.2.1. Via the Admin UI

The most common path for operators. Navigate to **Settings → Custom Fields → Select Entity → Add Field**:

1. Choose the entity (Invoice, Customer, Product, etc.)
2. Enter the field key (immutable after creation, must be snake_case, no spaces)
3. Enter the display label (mutable, translated per locale)
4. Choose the field type
5. Configure type-specific options (select options, link target entity, etc.)
6. Set validation rules (required, max length, regex, etc.)
7. Set the section and display order
8. Save — the field is immediately available in all forms and API responses

The UI performs a server-side preview of the field's rendered form component before saving.

### 10.2.2. Via the API

For programmatic field creation (tenant onboarding scripts, module installers):

```bash
POST /api/v1/custom-field-defs
Authorization: Bearer {tenant_admin_token}
Content-Type: application/json

{
  "entity_name": "Invoice",
  "field_key": "kra_customs_code",
  "label": "KRA Customs Classification",
  "field_type": "data",
  "required": false,
  "validation_rules": {
    "max_length": 20,
    "regex": "^HS-[0-9]{4}$"
  },
  "sort_order": 100,
  "section_label": "Tax & Compliance"
}
```

### 10.2.3. Via Fixture Files — For Module Developers

Module developers shipping pre-configured field sets include a `custom_fields.yaml` fixture:

```yaml
# modules/kenya-compliance/fixtures/custom_fields.yaml
entity: Invoice
fields:
  - field_key: kra_customs_code
    label: "KRA Customs Code"
    field_type: data
    required: false
    validation_rules:
      max_length: 20
  - field_key: etims_cu_serial
    label: "eTIMS CU Serial"
    field_type: data
    required: true
    validation_rules:
      max_length: 50
```

These fixtures are loaded during module installation (`awo module install kenya-compliance`) and are applied per-tenant during onboarding.

### 10.2.4. Supported Field Types for Custom Fields

Custom fields support a subset of the full field type list. Types not supported for custom fields:

| Type | Reason not supported |
|---|---|
| `LongText` | Performance: large text in JSONB is inefficient |
| `Table` | Requires a separate child entity, not expressible in JSONB |
| `AttachImage` | Binary metadata management not feasible in JSONB |
| `UUID` | No automatic generation in JSONB context |

All scalar types, `Select`, `MultiSelect`, and `Link` are supported.

## 10.3. Validation Rules on Custom Fields

### 10.3.1. Declared Validation Rules in `CustomFieldDef`

```go
type ValidationRules struct {
    Required  bool    `json:"required"`
    MinLength int     `json:"min_length,omitempty"`
    MaxLength int     `json:"max_length,omitempty"`
    Min       *float64 `json:"min,omitempty"`
    Max       *float64 `json:"max,omitempty"`
    Regex     string  `json:"regex,omitempty"`
    // e.g. "^[A-Z0-9]{10}$" for a membership number format
}
```

### 10.3.2. How Custom Field Validators Are Loaded and Executed

At request time, the framework:

1. Reads all `CustomFieldDef` records for the entity from cache (TTL: 60 seconds per tenant)
2. Extracts the `custom_fields` values from the incoming payload
3. For each value present, finds the matching `CustomFieldDef` and applies its validation rules
4. For each `CustomFieldDef` with `required: true`, checks that a value is present
5. Returns all errors before touching the database

```go
func validateCustomFields(ctx context.Context, defs []CustomFieldDef, values map[string]interface{}) []validate.FieldError {
    var errs []validate.FieldError
    defsByKey := indexDefsByKey(defs)

    // Check required fields
    for _, def := range defs {
        if def.Required {
            if _, ok := values[def.FieldKey]; !ok {
                errs = append(errs, validate.FieldError{
                    Field:   "custom_fields." + def.FieldKey,
                    Message: fmt.Sprintf("%s is required", def.Label),
                })
            }
        }
    }

    // Validate provided values
    for key, value := range values {
        def, ok := defsByKey[key]
        if !ok {
            // Unknown custom field key — reject to prevent key injection
            errs = append(errs, validate.FieldError{
                Field:   "custom_fields." + key,
                Message: "unrecognised custom field",
            })
            continue
        }
        if fieldErrs := validateCustomFieldValue(def, value); len(fieldErrs) > 0 {
            errs = append(errs, fieldErrs...)
        }
    }

    return errs
}
```

Unknown custom field keys are rejected. This prevents tenants from storing arbitrary data in the `custom_fields` object beyond what they have explicitly defined.

### 10.3.3. Validation Error Reporting

Custom field validation errors use the dot-path format to allow amis to bind them to the correct form field:

```json
{
  "errors": [
    { "field": "custom_fields.kra_customs_code", "message": "must match format HS-NNNN" },
    { "field": "custom_fields.insurer_name", "message": "Insurer name is required" }
  ]
}
```

The amis form uses nested field paths for custom fields. The form field's `name` property is set to `custom_fields.${field_key}` to match the error binding.

## 10.4. Querying Custom Fields

### 10.4.1. JSONB Path Predicates in the Filter DSL

```go
// Find invoices where kra_customs_code = "HS-8471"
invoices, _, err := repo.Query(ctx,
    filter.JSONPath("custom_fields", "kra_customs_code", filter.Eq("", "HS-8471")),
)

// Find invoices where insurer is Britam
invoices, _, err := repo.Query(ctx,
    filter.JSONContains("custom_fields", map[string]interface{}{
        "insurer_name": "Britam",
    }),
)

// Multiple custom field conditions
invoices, _, err := repo.Query(ctx,
    filter.JSONPath("custom_fields", "requires_pre_auth", filter.Eq("", true)).
    And(filter.JSONPath("custom_fields", "insurer_name", filter.Eq("", "AAR"))),
)
```

### 10.4.2. GIN Index Strategy — Which Paths to Index

The default GIN index on `custom_fields` supports containment queries (`@>`) efficiently. For equality queries on specific paths (`->>'key' = 'value'`), a path-specific expression index is needed:

```sql
-- Created when tenant marks "kra_customs_code" as indexed in admin UI
CREATE INDEX ON invoices ((custom_fields->>'kra_customs_code'));
```

The framework creates this index automatically when the tenant enables indexing on a `CustomFieldDef`. The index name follows the convention:
`idx_{table}_{entity_name}_{field_key}`.

Index creation uses `CREATE INDEX CONCURRENTLY` so it does not lock the table during creation.

### 10.4.3. Performance Characteristics

| Query pattern | Indexed? | Performance |
|---|---|---|
| `custom_fields @> '{"key": "val"}'` | GIN on column | Good — GIN index used |
| `custom_fields->>'key' = 'val'` | Expression index on path | Good if index exists |
| `custom_fields->>'key' = 'val'` | No expression index | Sequential scan — AVOID |
| `custom_fields->>'key' LIKE '%val%'` | Any | Sequential scan — always |
| Sorting by custom field | Any | Sequential scan — avoid in large tables |

When a custom field is being queried by hundreds of concurrent users (e.g. the KRA customs code on invoices for a tax reporting workflow), always enable indexing in the admin UI. When a custom field is rarely queried, the overhead of maintaining the expression index is not worth it.

**When to promote a custom field to a system column**: If a custom field:
- Is queried by >80% of API calls to the entity
- Is sorted or aggregated regularly
- Has >10M rows in the table
- Is needed in a JOIN with another table

then write a migration to promote it to a proper SQL column (see §10.6.5).

## 10.5. Surfacing Custom Fields in the SDUI Layer

### 10.5.1. Page Builder Reads `CustomFieldDef` Records at Render Time

When the SDUI layer generates a form or list view for an entity, it reads the entity's `CustomFieldDef` records and injects the custom fields into the schema dynamically:

```go
func BuildInvoiceForm(ctx context.Context, cfg *FormConfig) (*amis.Form, error) {
    // Get standard system fields
    form := standardInvoiceForm(cfg)

    // Load custom field definitions for this tenant
    defs, err := customFieldRepo.Query(ctx,
        filter.Eq("entity_name", "Invoice").
        And(filter.IsNull("deprecated_at")),
        query.OrderBy("section_label", query.Asc),
        query.OrderBy("sort_order", query.Asc),
    )
    if err != nil {
        return nil, err
    }

    // Group by section
    sections := groupDefsBySection(defs)
    for sectionLabel, sectionDefs := range sections {
        section := amis.NewFieldset(sectionLabel)
        for _, def := range sectionDefs {
            section.AddField(buildCustomFormField(def))
        }
        form.AddSection(section)
    }

    return form, nil
}
```

### 10.5.2. Automatic Form Field Injection — Ordering and Section Placement

Custom fields are injected after all system fields by default. Within custom fields, ordering follows:
1. `section_label` (alphabetical, sections without labels appear last)
2. `sort_order` within each section (ascending, lower numbers appear first)

The page builder respects the tenant's configured section labels. If a tenant sets `section_label: "Tax & Compliance"` on three custom fields, those three fields appear together under a "Tax & Compliance" heading in the form, separate from the standard fields.

### 10.5.3. Tenant Control Over Field Placement via UI Metadata

Tenants can also control exactly which system field a custom field appears after, via `UIMetadata`:

```go
type UIMetadata struct {
    PlacementAfterField string `json:"placement_after_field"` // system field name
    Width               string `json:"width"`                 // "full", "half", "third"
    HelpText            string `json:"help_text"`
    Placeholder         string `json:"placeholder"`
    Visible             string `json:"visible"`               // amis condition expression
}
```

Example: place the `kra_customs_code` field immediately after the `total_amount` field:

```json
{
  "placement_after_field": "total_amount",
  "width": "half",
  "help_text": "Enter the HS code for customs classification. Required for imports over KES 50,000."
}
```

## 10.6. Custom Field Lifecycle

### 10.6.1. Adding a Custom Field — No Migration Required for System Entities

Adding a custom field does not require a database migration. The `custom_fields` JSONB column already exists. The new `CustomFieldDef` record is created, and the field immediately appears in forms and is available in the API.

For the first time an entity has custom fields, the `custom_fields` column must exist. This is created by the system entity migration when the entity is first deployed (not when the first custom field is added).

### 10.6.2. Renaming a Custom Field — Key Stability vs Label Changes

**Label changes** (what the user sees in the form) are always safe. Update the `label` column in `CustomFieldDef`. No data changes required.

**Key renames** (the JSON key in `custom_fields`) are breaking changes:
- All existing data continues to use the old key
- New data will use the new key (if you update the `field_key` column — which is declared `Immutable`)
- Queries filtering by the old key stop returning data
- API callers that reference the key in their integration code will break

The `field_key` column is declared `Immutable` to prevent accidental key renames. To rename a key:

1. Create a new `CustomFieldDef` with the new key
2. Run a data migration to copy values: `UPDATE invoices SET custom_fields = custom_fields || jsonb_build_object('new_key', custom_fields->>'old_key') - 'old_key'`
3. Deprecate the old `CustomFieldDef` (set `deprecated_at`)
4. Remove the old key after a migration period

### 10.6.3. Changing a Field Type — Data Migration Implications

Field type changes (e.g. from `data` to `select`) are not directly supported. The `field_type` column is `Immutable`. Instead:

1. Create a new `CustomFieldDef` with the new type and a different key
2. Write a data migration that transforms existing values
3. Deprecate the old field
4. Update all API integrations and UI references

Never change the stored JSON values without an explicit data migration script that is reviewed and tested. Silent data corruption from a type change is extremely difficult to detect after the fact.

### 10.6.4. Deprecating and Removing a Custom Field — Soft Removal First

```go
// Set deprecated_at to stop the field from appearing in forms and API responses
def, _ := customFieldDefRepo.Update(ctx, defID, CustomFieldDefUpdate{
    DeprecatedAt: ptr.Time(time.Now()),
})
```

After deprecation:
- The field no longer appears in forms or list views
- API responses no longer include the field key in `custom_fields`
- Existing data values remain in the JSONB column
- Queries using the deprecated field key return no results (the framework silently ignores unknown keys)

Hard removal (deleting the `CustomFieldDef` and scrubbing the data) is a separate, explicit step that requires a data migration:

```sql
-- Remove deprecated field from all invoice records
UPDATE invoices SET custom_fields = custom_fields - 'deprecated_field_key';
```

Retain the `CustomFieldDef` record even after scrubbing, with `deprecated_at` set, for audit trail purposes.

### 10.6.5. Promoting a Custom Field to a System Field — When and How

When a custom field has proven its value and is used heavily across all tenants, it is a candidate for promotion to a system field:

1. **Add the column to the ent schema**:
```go
field.String("kra_customs_code").
    MaxLen(20).
    Optional().
    Comment("Promoted from custom_fields in migration 002001")
```

2. **Generate and run the migration**:
```sql
ALTER TABLE invoices ADD COLUMN kra_customs_code varchar(20);
UPDATE invoices SET kra_customs_code = custom_fields->>'kra_customs_code'
WHERE custom_fields ? 'kra_customs_code';
ALTER TABLE invoices ALTER COLUMN kra_customs_code SET NOT NULL WHERE condition...
```

3. **Dual-write for one release**: The framework writes to both the new column and the old custom_fields key for one release cycle.

4. **Deprecate the CustomFieldDef**: After all tenants have migrated.

5. **Remove from custom_fields**: Clean up the JSONB column.

Promotion is a multi-release operation. Never do it in a single migration — the risk of breaking existing API integrations and UI configurations is too high.
