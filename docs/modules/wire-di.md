# Wire & Dependency Injection

Awo ERP uses [Google Wire](https://github.com/google/wire) for compile-time dependency injection. Wire reads provider declarations, resolves the full dependency graph, and generates `wire_gen.go` — no reflection, no runtime overhead.

---

## File Map

```
internal/platform/wire/
  providers.go   — provider set declarations (the "index")
  platform.go    — platform-layer constructors (DB, cache, logger, metrics, tracing, temporal)
  services.go    — business-logic constructors (tenant, finance, ...)
  api.go         — API-layer constructors (Fiber, middleware, IAM, audit, API keys, handlers, router)

cmd/server/
  wire.go        — injector function (build tag: wireinject)  ← Wire reads this
  wire_gen.go    — generated wiring code (build tag: !wireinject)  ← runs in production
```

---

## Provider Set Hierarchy

```
ApplicationProviderSet          (providers.go)
├── PlatformProviderSet         config, DB, cache, logger, metrics, tracing, temporal
├── RepositoryProviderSet       (currently empty — repos live in CoreServiceProviderSet)
├── CoreServiceProviderSet      tenant, IAM, session, audit, API keys, finance
└── APIProviderSet              Fiber app, middleware, handler deps, router
```

`ApplicationProviderSet` is the only set passed to `wire.Build()`. Wire transitively includes everything else.

---

## How Wire Works Here

1. `cmd/server/wire.go` declares the injector:

   ```go
   //go:build wireinject

   func InitializeApplication() (*Application, error) {
       wire.Build(wirepkg.ApplicationProviderSet, NewApplication)
       return &Application{}, nil // placeholder, never compiled
   }
   ```

2. You run `wire` (or `go generate`) from `cmd/server/`:

   ```bash
   cd cmd/server && wire
   # or
   go generate ./cmd/server/...
   ```

3. Wire writes `cmd/server/wire_gen.go` with the full resolved construction sequence. **Do not edit this file manually.**

4. Normal builds use `wire_gen.go` (via `//go:build !wireinject`). The `wire.go` placeholder is excluded.

## Adding a New Module — Step by Step

Use the **finance module** shape as a reference. Follow these steps every time.

### Step 1 — Define interfaces in your domain package

```go
// internal/core/invoice/service.go
package invoice

type Repository interface {
    Create(ctx context.Context, inv *Invoice) error
    GetByID(ctx context.Context, id uuid.UUID) (*Invoice, error)
}

type Service interface {
    Issue(ctx context.Context, req IssueRequest) (*Invoice, error)
}
```

### Step 2 — Implement the constructors

```go
// internal/core/invoice/repository.go
func NewRepository(store db.Store, log logger.Logger, tracer tracing.Service) Repository {
    return &pgRepository{store: store, log: log, tracer: tracer}
}

// internal/core/invoice/service_impl.go
func NewService(repo Repository, cache cache.Service, log logger.Logger) Service {
    return &service{repo: repo, cache: cache, log: log}
}
```

Constructors must:
- Accept only types that Wire already knows how to provide (or that you'll register).
- Return the interface type (not the concrete struct), or a concrete type if there's no interface.
- Return `(T, error)` if construction can fail; return just `T` if it can't.

### Step 3 — Write provider functions in `internal/platform/wire/services.go`

```go
// NewInvoiceRepository constructs the invoice repository.
func NewInvoiceRepository(store db.Store, log logger.Logger, tracer tracing.Service) invoice.Repository {
    return invoice.NewRepository(store, log, tracer)
}

// NewInvoiceService constructs the invoice service.
func NewInvoiceService(repo invoice.Repository, cache cache.Service, log logger.Logger) invoice.Service {
    return invoice.NewService(repo, cache, log)
}
```

Wire matches providers by **return type**. Each type may only have one provider in the graph.

### Step 4 — Register in the appropriate provider set (`providers.go`)

```go
var CoreServiceProviderSet = wire.NewSet(
    // ... existing providers ...

    // Invoice
    NewInvoiceRepository,
    NewInvoiceService,
)
```

Order within the set doesn't matter — Wire resolves the order itself.

### Step 5 — Expose via `handlers.Dependencies` if needed

If the service needs to be reachable from HTTP handlers:

```go
// internal/api/handlers/deps.go
type Dependencies struct {
    // ... existing fields ...
    InvoiceService invoice.Service
}
```

Then add it to the `NewHandlerDependencies` provider in `internal/platform/wire/api.go`:

```go
func NewHandlerDependencies(
    // ... existing params ...
    invoiceSvc invoice.Service,
) *handlers.Dependencies {
    return &handlers.Dependencies{
        // ... existing fields ...
        InvoiceService: invoiceSvc,
    }
}
```

### Step 6 — Regenerate

```bash
cd cmd/server && wire
```

Wire will fail with a clear error if:
- A dependency type has no provider.
- A type has two providers (ambiguous).
- There's a dependency cycle.

Fix the error, then regenerate. **Never edit `wire_gen.go` directly.**

## Common Patterns

### Provider that can fail

```go
func NewAuthzService(store db.Store, ...) (iam.AuthzService, error) {
    svc, err := iam.New(iam.Config{...})
    if err != nil {
        return nil, fmt.Errorf("authz init: %w", err)
    }
    return svc, nil
}
```

Wire propagates the error up through `InitializeApplication() (*Application, error)`.

### Provider that reads from config

```go
func NewSessionService(
    repo iam.SessionRepository,
    cfg *config.Config,
    log logger.Logger,
) iam.SessionService {
    return iam.NewSessionServiceWithConfig(repo, log, iam.SessionConfig{
        SessionTTL: cfg.Auth.SessionTTL,
        CookieName: cfg.Auth.CookieName,
    })
}
```

`*config.Config` is already provided by `config.Load` in `PlatformProviderSet`, so Wire injects it automatically.

### Grouping multiple services into a struct

When a domain has several closely related services, group them:

```go
// internal/core/finance/service/services.go
type Services struct {
    Invoice invoice.Service
    Payment payment.Service
}
```

```go
// internal/platform/wire/services.go
func NewFinanceServices(store db.Store, log logger.Logger, ...) *financeService.Services {
    return &financeService.Services{
        Invoice: invoice.NewService(...),
        Payment: payment.NewService(...),
    }
}
```

Wire treats `*financeService.Services` as a single node in the graph.

## Rules & Constraints

| Rule | Reason |
|------|--------|
| One provider per type | Wire errors on ambiguity |
| Constructors take interfaces, not concrete types | Keeps layers decoupled and testable |
| Never edit `wire_gen.go` | It is overwritten on next `wire` run |
| Run `wire` after every provider change | Stale `wire_gen.go` causes compile errors |
| Providers belong in the layer file that matches their layer | `platform.go` → infra, `services.go` → business logic, `api.go` → HTTP |
| `RepositoryProviderSet` is currently empty | Repos are registered in `CoreServiceProviderSet`; move them if you want strict layer separation |

## Known Issues

| Issue | Impact |
|-------|--------|
| `NewFinanceServices` returns `nil` | Any handler using `FinanceServices` will panic — fix before wiring real finance handlers |
| `NewRedisClient` in `platform.go` is unused | Dead code; `NewCacheService` creates its own client internally |
| `NewTenantTemporalIntegration` output has no consumer | Wire may warn; its side-effects (worker registration) are lost |
| No `wire.Cleanup` anywhere | DB pool, Redis, Temporal client, and tracing exporter are not closed on shutdown — handle in `main.go` signal handler |
